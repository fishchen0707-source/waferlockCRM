## 2. 專案背景與目的

### 2.1 背景
WAFERLOCK（電子鎖品牌）目前營運一套售後服務 CRM 系統，現況為**雲端架構**（前端靜態網頁 + Supabase/PostgreSQL 雲端後端），透過網際網路運作。

### 2.2 委建目的
基於下列考量，需求方決定**委外重新建置一套部署於公司內部網路（內網）的版本，不再依賴雲端 Supabase 服務**：

1. **資料自主**：客戶個資、工單、照片等資料改由公司自行掌控於內網。
2. **資安顧慮**：現場師傅若以 VPN 連入內網存取系統，存在憑證外洩、帳號被盜、離職未撤銷等內網遭竄改風險。
3. **長期營運**：降低對單一雲端供應商的依賴。

### 2.3 核心設計原則（資安底線，不可妥協）
- **現場師傅一律不發 VPN**。師傅端與 CRM 端皆透過「對外公開的 HTTPS 服務 + 嚴格身份驗證」存取，不直接進入內網。
- **內網資料庫與內部服務完全不對外**，僅由受控的 API 層存取。
- **最小權限原則**：每個角色只能存取其職務所需的最少資料。

## 4. 現況系統架構（遷移前）

### 4.1 架構概觀
```
  [師傅手機瀏覽器]            [雲端 Supabase]              [CRM 瀏覽器]
  waferlock_tech.html  ──REST/WebSocket──  PostgreSQL  ──REST/WebSocket──  waferlock_crm.html
                                            + Realtime
                                            + 物件儲存(work-photos)
                                            + Edge Functions(LINE/Meta webhook)
```

### 4.2 技術現況
- **前端**：3 個獨立的靜態 HTML 檔（無打包工具），以瀏覽器 CDN 載入 React 18 + Babel Standalone 即時編譯。
  - `waferlock_crm.html` — CRM 客服後台（**本次遷移**）
  - `waferlock_tech.html` — 師傅端（**本次遷移**）
  - `waferlock_LINE.html` — LINE/FB/IG 客服平台（**本次不遷移**）
- **後端**：Supabase（PostgreSQL）。前端**直接**呼叫 Supabase REST API 與 Realtime（WebSocket）。
- **即時同步**：透過 Supabase Realtime，CRM 與師傅端跨裝置即時同步。
- **檔案儲存**：施工/簽收照片存於 Supabase Storage 的 `work-photos` bucket。
- **外部串接**：LINE / Facebook / Instagram 訊息透過 Supabase Edge Functions（webhook + push）收發。

### 4.3 現況已知資安債（遷移後**必須改善**）
- 🔴 **密碼明文**：使用者密碼（`crm_users.pin`、`workers.pin`）以明文儲存，預設值皆為 `1234`。
- 🔴 **無後端權限控管**：資料庫權限政策全開（任何持金鑰者可讀寫所有資料），登入僅為前端畫面閘門，非真正的後端授權。
- 🔴 **金鑰硬編碼於前端**：連線資訊寫死在 HTML 內。

> 廠商**必須**在新系統中根治以上三點（見第 10、11 章）。

## 5. 目標系統架構（遷移後）

### 5.1 架構概觀
```
  公司防火牆外（網際網路）                 │ 公司內網（不對外）
                                          │
  [師傅手機]                              │
  師傅端 ───┐                             │
            ├──HTTPS(443)──▶ [API 層]  ──▶ [PostgreSQL 資料庫]
  [CRM 端]  │              (唯一對外開口)  ──▶ [檔案/物件儲存]
  CRM ──────┘                  │          │   └─ 內部服務皆不對外
                          身份驗證(JWT)    │
                          權限控管/log     │
```

### 5.2 網路分層要求
1. **對外只開放一個 HTTPS 端點（443）**，其餘連接埠對外一律封閉。
2. **API 層為唯一對外開口**：負責身份驗證、權限控管、資料存取。
3. **資料庫、檔案儲存、其他內部服務完全不對外**，僅 API 層可存取。
4. **師傅端與 CRM 端皆走公開 HTTPS**，**不需 VPN、不直接連內網資料庫**。
5. 建議於 DMZ（非軍事區）部署 API 層／反向代理，內網資料層再退一層。

### 5.3 部署環境
- 部署於公司內部伺服器（實體機或虛擬機），具固定對外網域 + 有效 TLS 憑證。
- 作業系統、資料庫版本、容器化與否由廠商提案，需於文件載明。

## 7. 功能需求（依模組）

> 以下為現有系統已實作、需於新系統**完整重現**的功能。標示 ☐ 供驗收勾選。

### 7.1 CRM 客服後台

**客戶與案件**
- ☐ 客戶主檔管理（新增/編輯/查詢），客編（WF）以「地址 + 姓名」為唯一鍵
- ☐ 首頁案件大池 + 客戶導向搜尋（搜尋 → 客戶完整詳情頁）
- ☐ 集團 → 建案兩層階層瀏覽，客戶可歸屬建案

**工單**
- ☐ 統一工單表單：單一表單建立安裝 / 維修單，依類別動態顯示欄位
- ☐ 客訴單建立、追蹤（含下次追蹤時間）、結案、轉派工
- ☐ 收費品項表（品名 × 數量 × 單價 + 贈品，自動合計）
- ☐ 付款資訊記錄（付款方式 現金/轉帳/刷卡/免費、收款狀態、交易編號、收款時間）
- ☐ 保固判定（保內/保外/保內不保固 + 原因）
- ☐ 完工驗收電子工單（照片、簽名、收費明細）

**派工與排程**
- ☐ 派工排程：選區域 / 師傅 / 時段，佔用時段格（維修佔 1 格、安裝佔 2 格、可變工時 = 無條件進位佔格）
- ☐ 工務時刻表：分區（北/中/南）+ 每日 13 格時段視圖，格子顯示案件編號、點擊看詳情
- ☐ 時段衝突檢查

**帳號 / 權限 / 稽核**
- ☐ 帳密登入；管理員可切換部門視角，一般員工固定看本部門
- ☐ 登入者姓名背景浮水印（責任歸屬）
- ☐ 案件異動紀錄：派工/結案/建單/排程/施工/驗收皆記錄「時間・動作・操作者」；入口為工單卡片上的低調小按鈕（非常駐顯示），點擊才展開/收合，避免干擾一般操作視覺

**地址**
- ☐ 中華郵政風格地址選取器（縣市 → 鄉鎮 → 郵遞區號自動帶入）

**QC / 報表（2026-06-30 新增）**
- ☐ 維修結案三階段 QC 分類：型號確認 → 故障原因（大類 + 依型號真實屬性動態組出的細項）→ 處理過程，取代原本自由文字備註，供 QC 依型號統計故障率
- ☐ 建單/派工階段設定的收費品項，須帶入師傅端結案畫面作為初始值（可編輯/追加，非唯讀），不可要求師傅重複輸入
- ☐ 師傅報表（管理端）：可切換師傅＋月份，檢視案件數、應收帳款（已收/待收分列）、平均處理時效、型號故障率排行、未收款/待對帳清單、工單細項

### 7.2 師傅端

- ☐ 工班帳密登入
- ☐ 接案 / 拒接
- ☐ ETA 抵達時間回推：接案時讀取定位 → 計算與客戶地址距離 → 依個人歷史校準 → 回推預計抵達時間範圍（抓不到定位時可手動選時間範圍）
- ☐ 到場 / 施工中 / 施工完成 狀態回報
- ☐ 施工 / 簽收照片上傳、客戶簽名
- ☐ 收費品項與付款方式登錄、結案
- ☐ 安裝完工 → 觸發保固登錄
- ☐ 「無法安裝 / 無法維修」回報
- ☐ 師傅僅能看到/操作**指派給自己**的工單（見第 10 章權限）
- ☐ 結案三階段 QC 分類填寫（型號確認、故障原因大類+細項、處理過程）
- ☐ 「我的報表」：登入師傅可查自己當月案件數、應收帳款、平均處理時效、型號故障率排行、未收款清單

### 8.3 資料庫邏輯（RPC / 函式）
- ☐ **案件編號產生器**：現有 `next_case_no(prefix)` 產生「類別碼 + YYYYMMDD + 4 碼日流水」（每日每類重置、**原子遞增不重號**）。新系統須等效實作，保證高併發下不重號。

### 8.4 檔案 / 照片儲存
- 現況：施工/簽收照片存於雲端物件儲存（`work-photos`），資料表只存檔案 URL。
- 新系統須提供**內網檔案/物件儲存方案**（如 MinIO 或檔案系統 + 受控存取），廠商提案。
- 要求：照片不得以 base64 直接塞入資料列（避免效能問題）；存取須經權限控管，不可公開匿名存取。

## 9. 即時同步需求

- CRM 端須**即時反映**師傅端的狀態變更（例：師傅按「施工完成」，CRM 客服畫面即時更新，無需手動重整）。
- 現況以 Supabase Realtime（WebSocket）達成；新系統須提供等效機制，技術方案（**WebSocket** 或 **Server-Sent Events**）由廠商提案。
- ☐ 驗收標準：A 端資料變更後，B 端畫面於 **3 秒內**自動更新。

## 10. 身份驗證與權限需求

### 10.1 身份驗證
- ☐ 以 **JWT token** 為基礎的登入機制（取代現有前端假登入）。
- ☐ Token 有效期上限（建議 ≤ 8 小時），過期須重新登入。
- ☐ 登入失敗鎖定 / 防暴力破解機制。

### 10.2 角色與權限
| 角色 | 權限範圍 |
|------|---------|
| 管理員（admin） | 全系統，可管理帳號、切換部門視角 |
| 部門客服／業務 | 依部門檢視客訴等資料 |
| 師傅（工班） | **僅能存取指派給自己的工單**，看不到其他師傅、看不到非相關客戶完整資料 |

- ☐ 權限於**後端 API 層強制**（非僅前端隱藏），未授權請求一律於後端拒絕。
- ☐ 管理員可**即時停用/撤銷**任一帳號（離職、遺失裝置）。

### 10.3 密碼安全
- ☐ 密碼一律**雜湊儲存**（如 bcrypt/argon2），**嚴禁明文**。
- ☐ 不得有共用預設密碼；首次登入或建立帳號須設定強密碼。

## 11. 資安需求

- ☐ **對外僅開放 443（HTTPS）**，其餘連接埠對外封閉。
- ☐ 資料庫、內部服務不對外，僅 API 層可存取。
- ☐ 全程 TLS 加密；有效憑證、自動續期。
- ☐ **密鑰 / 連線字串以環境變數或祕密管理工具保存，嚴禁硬編碼於程式或前端。**
- ☐ **操作稽核 log**：記錄「誰、何時、對哪筆資料、做了什麼」，保留 **≥ 90 天**。
- ☐ **自動備份**：資料庫每日自動備份，保留 **≥ 30 天**；照片/檔案一併納入備份。
- ☐ **還原演練**：交付前須實際演練一次備份還原並提供報告。
- ☐ 個資存取最小化、傳輸與靜態加密（依公司 PDPA 政策）。
- ☐ 上線前提供**弱點掃描報告**（至少涵蓋 OWASP Top 10 常見項目）。

## 13. 資料落地方案（請廠商評估報價）

> 需求方尚未決定，請廠商就以下兩案分別說明可行性、成本與優缺點：

### 方案 A — 全部留內網
- DB、照片、所有資料皆不出公司網路。
- 優點：資安等級最高、資料完全自主。
- 缺點：未來要接外部平台（LINE/FB/IG webhook 需公開端點）時架構較複雜。

### 方案 B — 核心內網 + 外部串接走 DMZ/中繼（建議）
- CRM、資料庫、照片於內網；未來對外 webhook 經 DMZ 或受控雲端中繼轉入。
- 優點：兼顧資安與外部串接可行性。
- 缺點：架構分層較多，需妥善設定防火牆規則。

## 14. 非功能需求

- ☐ **效能**：CRM 首頁載入 ≤ 3 秒；一般操作回應 ≤ 1 秒（內網環境）。
- ☐ **可用性**：營業時間可用率 ≥ 99%；提供基本監控與告警。
- ☐ **瀏覽器相容**：支援最新版 Chrome / Edge（CRM 端）。
- ☐ **行動裝置**：師傅端須在主流手機瀏覽器（Android/iOS）正常操作（含定位、拍照、簽名）。
